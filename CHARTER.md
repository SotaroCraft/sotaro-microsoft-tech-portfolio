# MicroBootCan プロジェクト憲章

Azure フルスタック構成の個人向け PWA。実用性とクラウドネイティブのポートフォリオ価値を両立する。

> 実装計画の詳細は [MVP Plan](../../.cursor/plans/microbootcan_mvp_plan_3560032d.plan.md) を参照。

---

## ミッション

**目標の可視化・構造化ジャーナル・実績サマリー・オポチュニティ管理** を一つのツールに集約し、日常の規律と振り返りを支援する。同時に、**Azure ネイティブ構成の実践例** として公開する。

---

## 公開コピー原則（必須）

GitHub・デモ URL・README・ランディング（`/`）など **第三者が読めるすべての文章** は、個人の目的や背景を推測させない中立表現に統一する。

### 公開で使う言い方

| 概念（内部） | 公開向けの呼び方 |
|-------------|-----------------|
| カウントダウン | **Milestone countdown** / マイルストーン・カウントダウン |
| 構造化ジャーナル（STAR 形式） | **Achievement journal** / 実績ジャーナル |
| 実績サマリー | **Metrics & summary** / メトリクスとサマリー |
| オポチュニティ管理 | **Pipeline tracker** / パイプライン管理 |
| テキストマッチング | **Context matching** / コンテキスト・マッチング |
| 参照テキスト貼付 | **Reference text input** / 参照テキスト入力 |

### 公開で使わない言い方

転職、選考、面接、応募、補助金、危機感、ヘッドハンター、求人票、レジュメ、Recruiter Screen、Bar Raiser 等の **採用・転職文脈が明確な語**

### ルート別

| ルート | トーン |
|--------|--------|
| `/` | ポートフォリオ向け技術紹介 + 汎用プロダクト説明 |
| `/app/*` | 認証後の個人ワークスペース（中立ラベル。詳細文脈は UI 内カスタム可） |
| ソースコード | 公開リポジトリ内のコメント・README も上記に準拠 |

## 設計方針（優先順位）

1. **Azure ポートフォリオ映え** — 語れるクラウドネイティブ構成
2. **プライバシー保護** — 公開デモと個人データの厳格な分離
3. **コスト最適化（無料枠優先）** — 個人利用で実質 $0〜数ドル/月
4. **費用対効果の高い有料のみ例外** — Azure OpenAI 等、ROI が明確なもの

**月次予算上限**: **¥2,900 / $19.99**

---

## Azure コスト管理・承認原則

Azure リソースの構築・変更・操作に関する **エージェント・開発者共通の必須ルール**。Cursor エージェントは `.cursor/rules/azure-cost-governance.mdc` でも準拠する。

### 事前承認（必須）

**わずかでも費用が発生しうる変更・操作を行う前に、必ずユーザー承認を得る。**

### 事後報告（必須）

**Azure 操作を行った場合、毎回レスポンス末尾に「想定費用」を記載する。**

## 機能要件（MoSCoW）

### Must（MVP v0.1）

| 機能 | 目的 |
|------|------|
| **マイルストーン・カウントダウン** | 目標日（2027/4/1）までを秒単位で可視化し、期限意識を維持 |
| **実績ジャーナル** | 業務メモを構造化（Situation / Task / Action / Result）して蓄積・検索 |
| **メトリクス & サマリー** | 日英サマリー文 + 数値 KPI を常時最新に保つ |
| **パイプライン管理** | オポチュニティの進捗をカンバンで一元管理 |

### Should（v0.2 以降）

- 個人資産・キャッシュフローシミュレーター
- 住居コスト試算（賃貸 / 購入 / ローンタイミング）
- 市場ベンチマークモニター
- Microsoft 4色グリッドのダッシュボード UI

### Want（最終フェーズ）

- **ユーザー起点のデータ取り込み** — 参照テキスト貼付、CSV/JSON インポート、許可された公式 API / RSS
- **非公開ローカルスクレイパー** — 個人 PC のみ（後述）
- ジャーナルから日英ドキュメント PDF 自動生成

### Won't（公開範囲で非採用）

| 項目 | 理由 |
|------|------|
| **公開デプロイ上のスクレイピング** | ToS リスク・ポートフォリオ印象 |
| **個人データの公開露出** | Public リポジトリ・ランディング・スクリーンショットへの実データ混入 |
| **Electron デスクトップ** | Web + PWA で代替 |

## 公開 vs 非公開（プライバシー原則）

デモ URL はポートフォリオ用に **公開** する。個人データは **認証後の `/app/*` のみ**。

### 公開してよいもの

- アーキテクチャ図、技術スタック、Azure 構成
- UI の見た目（**ダミーデータのみ**のスクリーンショット）
- 「Entra ID で保護された個人ツール」という説明

### 公開・コミットしてはいけないもの

- 実名のジャーナルエントリ（会社名・プロジェクト名・数値）
- パイプライン上の実オポチュニティ名・連絡先
- 家計シミュレーションの具体数値
- サマリー文の本文
- **個人事情が読み取れる文脈**（ランディング・README・スクリーンショット）

### ルート別方針

| ルート | 認証 | 内容 |
|--------|------|------|
| `/` | 不要（公開） | 技術スタック、構成図、**ダミーデータ**のスクリーンショット |
| `/app/*` | Entra ID 必須 | Must 4機能（実データ） |
| `/api/*` | Entra トークン必須 | Functions API |

### 運用ルール

- デモ用シードは `packages/shared/demo-data/` の架空データのみ
- 公開ランディングのカウントダウンは **「Milestone: 2027-04-01」** 等の汎用表示
- README に Live Demo URL + 「owner-only authenticated workspace」

## 技術憲章（確定事項）

| 項目 | 決定 |
|------|------|
| フロント | Azure Static Web Apps + React PWA |

**ローカル開発**: [docs/local-dev.md](docs/local-dev.md)

## エンジニアリング原則

（アーキテクチャ・認証・Cosmos・AI・API・フロント・CI/CD・可観測性 — 従来どおり Azure ベストプラクティスに準拠）

| 領域 | 要点 |
|------|------|
| アーキテクチャ | サーバーレス、pnpm monorepo、Bicep IaC |
| 認証 | SWA Built-in Auth、シングルユーザー allowlist |
| データ | Cosmos `/userId` パーティション、Zod バリデーション |
| AI | ローカルはモック、`gpt-5-mini` + `text-embedding-3-small`、予算 ¥2,900 |
| フロント | Fluent UI、`/` 公開 + `/app/*` 認証必須 |
| ログ | PII・参照テキスト全文を Application Insights に出さない |

---

## データ取り込み方針

公開製品では **ユーザー起点の入力のみ**（テキスト貼付、CSV/JSON、許可された API/RSS）。Azure 上のサーバー側スクレイピングは Won't。ローカル非公開ツールは任意。
